Data Collection

The Oblique Approach

Collecting League of Legends match data requires a strategic approach due to Riot API's structure. We couldn't directly fetch match data - we needed to follow a multi-step process.

1. High Elo Player Collection

Started with Grandmaster and Challenger ranked leagues to get player PUUIDs from the first pages of each division

TIERS = ['GRANDMASTER', 'CHALLENGER']
DIVISIONS = ['I', 'II', 'III', 'IV']
url = f"https://kr.api.riotgames.com/lol/league-exp/v4/entries/{queue}/{tier}/{division}"

2. PUUID → Match IDs

For each player PUUID, fetched their recent ranked match IDs (20 matches per player)

url = f"https://asia.api.riotgames.com/lol/match/v5/matches/by-puuid/{puuid}/ids"
params={"queue": 420, "type": "ranked", "start": 0, "count": 20}

3. Match IDs → Full Match Data

Finally, fetched complete match information for each unique match ID

url = f"https://asia.api.riotgames.com/lol/match/v5/matches/{match_id}"

Rate Limiting & Optimization

We implemented sophisticated rate limiting and duplicate prevention to maximize API efficiency:

  • Smart Rate Limiting: Respected API headers and implemented exponential backoff
  • Duplicate Prevention: Saved 3,000+ API calls by detecting duplicate PUUIDs and match IDs
  • Checkpointing: Resumable data collection with graceful shutdown handling
  • Bucket Strategy: Processed data in batches to stay within API limits

Evolutionary Algorithm Comparison

Research Objective

Instead of traditional machine learning models, we implemented and compared two evolutionary algorithms to optimize League of Legends champion item builds for maximum win rate prediction.

Algorithm Selection Rationale

Evolutionary algorithms are ideal for this optimization problem because:

  • Discrete Search Space: Item builds are combinations of discrete items (not continuous variables)
  • Complex Interactions: Item synergies create non-linear relationships difficult for traditional ML
  • Multi-Objective: Optimizing for win rate while considering item cost and availability
  • No Gradient Information: Win rate function is not differentiable with respect to item choices

Genetic Algorithm (GA)

Approach: Binary encoding where each bit represents an item's presence/absence in the build

  • Population: 50 individuals
  • Generations: 100
  • Selection: Tournament selection
  • Crossover: Single-point crossover
  • Mutation: Bit-flip mutation

Differential Evolution (DE)

Approach: Continuous optimization with discrete mapping for item selection

  • Population: 50 individuals
  • Generations: 100
  • Strategy: DE/best/1/bin
  • Mutation Factor: 0.7
  • Crossover Rate: 0.8

Fitness Function Design

Both algorithms optimize the same objective: predicting win rate based on item builds using a Random Forest model trained on historical match data. The fitness function evaluates each candidate build against the trained model to predict its effectiveness.

Algorithm Evaluation and Validation

Evaluation Methodology

Our evaluation focused on comparing the effectiveness of Genetic Algorithm vs Differential Evolution for item build optimization, using multiple validation approaches.

Fitness Function Validation

Both algorithms used a Random Forest classifier as the fitness function, trained on historical match data to predict win probability based on item builds.

  • Training Data: 83,480 player records across 168 champions
  • Champion Focus: Champion 236 with 525 records (52.4% baseline win rate)
  • Feature Engineering: 7 item slots (item0-item6) as input features
  • Validation Split: 80/20 train-test split for model validation
